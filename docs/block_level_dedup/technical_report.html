Oxen · block-level deduplication

Store every version of your training data.

Training data changes constantly: sessions grow, prompts get revised, rows get relabeled. Oxen's block-v1 storage engine versions that data the way you version code — with compression that understands what the data is. Its first format-aware profile, built for agent LLM traces, stores 178 MB of six-commit trace history in 8.4 MB, byte-exact.

Different data deserves different compression

General-purpose storage treats every file as a byte stream. That is the right floor — any bytes, always safe — but it leaves the biggest wins on the table, because every format has its own edit grammar: the specific ways files of that kind actually change between versions. A chat session grows by appending messages. A checkpoint changes densely but low-rank. An embedding table appends rows and never edits them. A video timeline shuffles references to immutable clips.

Block-v1 is built so each of those can get its own pipeline. Chunkers, codecs, and transforms live behind append-only IDs in a registry; a policy — plus explicit per-path marks in your repo config — selects the pipeline per file. The choice is recorded in the manifest and is write-side only: readers reconstruct from manifests and never re-chunk, so a new profile ships without touching servers, old clients, or the wire protocol. Agent LLM traces are the first profile because that is where our users feel version-storage pain today. The same seam is waiting for the rest:

  • Model checkpoints & LoRA weights — tensor-aware deltas between fine-tuning runs, where consecutive checkpoints differ everywhere but shallowly.
  • Embeddings and vector dumps — float-array-aware chunking for .npy/safetensors/Arrow columns that append and re-index but rarely rewrite.
  • Logs, metrics, and event streams — the same append-heavy line grammar as traces with different tokens.
  • Genomics and scientific data — FASTA/FASTQ, sensor captures, point clouds: record-structured, reference-heavy, huge.
  • Creative assets — Ableton project files, OpenTimelineIO JSON timelines, 3D scene formats: structured documents that reference large immutable media and change by small structural edits.

Profile #1: agent LLM traces

Provider prompt caching exists because agent requests share a byte-identical prefix: tool definitions, then the system prompt, then the growing conversation — with exact-prefix matching and minimum cacheable sizes of 512–4,096 tokens (2–16 KB of text). The same structure dominates stored trace datasets:

  • Every session of an agent config repeats a verbatim 12–40 KB system-prompt + tools prefix.
  • Request-level logs re-serialize the entire conversation prefix on every row.
  • Live session logs are append-only event streams, re-uploaded as they grow.
  • A one-line prompt edit "invalidates" the prefix of every session that used it.

Generic content-defined chunking sees none of this. It cuts on byte statistics, so a chunk spans many rows and dies when any of them changes; per-chunk compression starts with an empty window, so a system prompt repeated 1,200 times compresses 1,200 times. We measured that failure precisely, then built the engine around the structure instead.

What runs when you oxen add

Content-adaptive chunking −16%

Line-delimited JSON sniffs its average row size (first 64 KB) and routes: long-trace rows get structural cuts at row and message boundaries — position-independent, so reordering costs nothing and an appended session re-stores only its tail. Small rows go through FastCDC, where 64 KB windows compress better than structure can save. The sniff is part of the frozen boundary function; the manifest records exactly which chunker ran.

Prompt-prefix isolation −4.4%

The storage twin of a cache breakpoint: any message-array element ≥ 4 KB is cut into its own chunk, separating the row's unique ID from the config's verbatim prefix. A thousand sessions of one agent config share one stored prefix chunk; a prompt rollout re-stores one chunk per config, not one per row.

Per-class shared dictionaries −10.5%

A 64 KB zstd dictionary per content family (long-row JSONL, small-row JSONL, CSV), trained on the first ingest of each class. It captures what no chunker can: repeated JSON keys, roles, tool names, and boilerplate scattered across rows. Class purity matters — one mixed dictionary measurably poisons both classes.

Bounded delta encoding −14.1%

Near-duplicate chunks are found by three positional sketches (head, middle, tail — the middle catches request-log rows whose shared tools+system block sits between a unique session ID and a fresh turn) and stored as a zstd delta against the base chunk — including bases from the same ingest pass, so the first snapshot's later rows delta against its earlier rows. Chain depth is capped at one; a delta is kept only when it is strictly smaller than every other encoding.

Underneath: chunks are content-addressed (xxh3-128), packed into immutable 64 MB blocks, and indexed in a rebuildable LMDB. Manifests are pure content — they never name block placement — and are themselves delta-compressed at rest against a content-lineage base. Everything store-local re-encodes to plain zstd on the wire, so servers and older clients interoperate unchanged.

Measured

Three fixed corpora, replayed as real oxen add/commit sequences against fresh repositories. Storage results are deterministic — repeat runs are byte-identical. zstd-19 each is the strongest naive alternative: compress every snapshot whole. It is genuinely hard to beat over a handful of commits; versioning advantages are slopes, and slopes need time. So we measured time:

By day 60 the gap is 254 MB against 30 MB and widening by ~3.9 MB per day: snapshot compression pays for the whole snapshot every day, oxen pays for what changed. Versioning the same file over time

This is the workload the engine is for — and the entire benchmark: every commit above re-presents the same evolving files (bigprompt_traces.jsonl five times, request_log.jsonl four, each live session log at 40 → 65 → 85 → 100% growth). Cumulative curves hide the number that compounds, so here it is directly: what the next version of your dataset costs to store.

Committing a new version of a ~30 MB dataset costs oxen 0.3–1.8 MB depending on the edit pattern — an append-heavy commit costs 1% of the snapshot, a prompt-rollout edit that touches a quarter of all rows costs 3%. Snapshot compression pays 0.9–2.9 MB for the same commits and retains none of the structure: no shared chunks to push incrementally, no partial fetch, and every restore decompresses a monolith.

The hardest baseline was zstd-19 itself. Whole-snapshot compression is brutally strong on this corpus: an 8 MB match window sees every repeated prefix at once, where a chunk-local encoder sees 64 KB. Mid-run it was genuinely ahead of us on the first snapshot (1.0 MB vs our 4.05), and we published that. Closing it took two specific mechanisms — a midpoint sketch, and same-pass delta bases so a snapshot's later rows can delta against its earlier rows during the first ingest. Final: 8.40 MB vs 8.5 on totals, 1.06 vs 1.42 MB on per-commit marginals — while keeping random access, partial fetch, and incremental push that a monolithic archive gives up.

Parquet: solved by a writer flag, bounded by arithmetic. Our first verdict was that compressed Parquet is a storage floor — writer-compressed zstd pages aren't byte-exactly reproducible by recompression, and reordered row groups leave ~0.2% of pages identical. That still holds for uncooperative writers. But Parquet's own content-defined chunking (pyarrow ≥ 21, use_content_defined_chunking=True — the approach described in Hugging Face's parquet-cdc work) makes page boundaries follow content, and byte-identical pages dedup through this engine unchanged; see the measurement below. The boundary we verified along the way: at ~500 rows per page, densely scattered row edits dirty every page in any layout — the same rows-per-chunk amplification law we measured for byte chunks — which is exactly the workload the JSONL trace profile exists for.

Same engine, same logical data, one writer flag: full-history storage drops 40%, a contiguous purge costs 5× less, a backfill 3× less. If you version Parquet, write it with CDC enabled; if your table takes dense scattered edits, version it as JSONL and export Parquet as a derived artifact.

How it scales

Storage cost separates into a one-time base (first snapshot, compressed) and a per-commit marginal — and the marginal is a per-byte property of the workload, not of corpus size. On the 60-day corpus the measured marginals are ~45 MB raw, ~4.3 MB for zstd-19 snapshots, and ~0.46 MB for oxen per daily commit. Amortized over history the base vanishes: sixty versions of the table cost half a megabyte each. Append-only workloads sit lower still (0.011× — a 29 MB append stored 0.32 MB).

This architecture was found by an agent

The trace profile was not designed on a whiteboard. We wrote a fixed benchmark — two deterministic multi-version corpora, byte-exact reconstruction as a hard gate, metrics ranked storage-first — and pointed Claude at it with a research charter: hypothesize, implement in the real engine, measure, keep only what wins, log everything including failures. It ran overnight, unattended.

Twenty experiments came out of the loop. The rules did real work: the tie threshold (0.5%) discarded a clever index redesign because it couldn't beat the simpler code it replaced; the lexicographic ordering kept zstd-19 despite a 3× ingest cost because storage outranks speed; the benchmark overruled the authors' favorite idea twice — the structural chunker lost to plain FastCDC on short-row data the moment dictionaries existed, which is why routing is adaptive rather than ideological. And the loop compounds: three of the strongest mechanisms (the 16 KB floor, midpoint sketches, in-flight delta bases) were found by chasing anomalies in earlier results, not by planning.

An agent optimizing storage for agent traces is a pleasing loop, but the consequence is practical: the method transfers. A new data type needs a corpus that mirrors how that data really evolves, and the same loop will find its profile — the harness, the comparison rules, and the research charter are all in this repository. If your datasets have structure this engine doesn't exploit yet, that is what pull requests are for.

What you can rely on

  • Byte-exact, always. Every read path reconstructs original bytes and verifies the whole-file hash. The benchmark treats any mismatch as total failure; none occurred.
  • History is untouched. Chunking lives below the Merkle tree — commit IDs, tree hashes, and file identities never change when storage format does. Migration runs both directions, resumable, with every version keeping at least one complete representation at all times.
  • Wire-compatible. Dictionary and delta codecs are store-local; transfer packing re-encodes to plain zstd. A server without them verifies pushes exactly as before.
  • Recoverable. The chunk index is derived state — delete it and oxen storage rebuild-index rebuilds it from self-describing block footers.
  • Frozen behavior. Chunkers, codecs, and thresholds are pinned behind append-only IDs with golden fixtures; a behavior change ships as a new ID, never a silent drift.
  • Extensible. The structural trace chunker is ~200 lines behind a trait. Adding a format-aware pipeline touches the registry and policy — manifests, blocks, the wire protocol, and old clients are untouched, because chunker choice is write-local.
